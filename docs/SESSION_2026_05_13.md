## Server Architecture: `ai-agents-x-space/index.js`

### Agent state machine
```javascript
state = {
  agents: {
    0: { id: 0, name: "Swarm", status: "offline", connected: false },
    1: { id: 1, name: "Swarm2", status: "offline", connected: false }
  },
  currentTurn: null,
  turnQueue: [],
  messages: [],    // last 100 messages
  isProcessing: false
}
```

`status` cycles: `offline` → `idle` → `listening` → `speaking` → `idle`

### Session endpoint
```javascript
GET /session/:agentId

// POSTs to:
"https://api.openai.com/v1/realtime/client_secrets"
// with model + voice + instructions
// Returns: { value: "ek_..." }  ← ephemeral key consumed by agent HTML page
```

### Turn management
`requestTurn(agentId)` — grants floor if idle, queues otherwise
`releaseTurn(agentId)` — frees floor, promotes next from queue

### Claim-token (prevents overtalking)
```javascript
if (status === "speaking") {
  // tell every OTHER connected agent to cancel their in-flight response
  other.socketId && io.to(other.socketId).emit("cancelResponse", { reason })
}
```

### Banter loop (two-agent)
```javascript
socket.on("textComplete", ({ agentId, text }) => {
  const other = state.agents[agentId === 0 ? 1 : 0]
  if (other.connected && other.socketId) {
    const sendWhenIdle = (attempt = 0) => {
      if (bothIdle) {
        io.to(other.socketId).emit("textToAgent", { text, from: sender.name })
      } else if (attempt < 10) {
        setTimeout(() => sendWhenIdle(attempt + 1), 1500)  // polls up to 15s
      }
      // else: drops silently
    }
    setTimeout(sendWhenIdle, 2000)  // initial delay for status update to land
  }
  // If other.connected === false → skipped entirely → single-agent mode
})
```

### Human passthrough
```javascript
socket.on("userMessage", () => io.emit("textToAgent", { text, from }))
// All STT transcripts from the Space flow here
```

### WebRTC connection flow in detail
```
1. GET /session/0
   → POST https://api.openai.com/v1/realtime/client_secrets
   → returns { value: "ek_..." }

2. new RTCPeerConnection({ iceServers: [stun:stun.l.google.com:19302] })

3. navigator.mediaDevices.getUserMedia({ audio: true })
   → Chrome finds PULSE_SOURCE from its env (agent1_mic)
   → captures audio from PulseAudio virtual source

4. pc.createDataChannel("oai-events")

5. pc.createOffer() → setLocalDescription

6. POST https://api.openai.com/v1/realtime/calls?model=gpt-4o-realtime-preview
   Authorization: Bearer ek_...
   Content-Type: application/sdp
   Body: offer.sdp
   → returns answer SDP

7. pc.setRemoteDescription(answer)

8. ICE connects → oniceconnectionstatechange = "connected"
   → socket.emit("agentConnect")

9. dc.onopen → send response.create with freestyle kick instruction
   → OpenAI starts generating → audio plays back through <audio> element
   → Chrome routes audio to PULSE_SINK (agent1_speakers)
   → PulseAudio routes to x_swarming_mic source
   → @swarminged Chrome picks up x_swarming_mic as its microphone
   → X Space WebRTC transmits it to listeners
```

### Audio analysis (VAD loop, runs every 33ms)
```javascript
SPEAK_THRESHOLD = 0.04   // volume level
SILENCE_DURATION = 500ms // silence before "stopped speaking"

// Reads frequency bins 3–25 (voice range)
// If level > threshold → status "speaking" → emit statusChange
// If level < threshold for 500ms → status "idle" → emit releaseTurn
```

### The greeting persona bug (discovered today, FIXED)
The `dc.onopen` sends a `response.create` with an override instruction. This fires on every connect and establishes the initial conversational context. It was still set to the old three.ws text:

## PulseAudio — Full Reference

### Why it must run as agent user
PulseAudio is a per-user daemon. It creates its socket at `$XDG_RUNTIME_DIR/pulse/native`. Chrome looks for it using the same path. If Chrome was launched without `XDG_RUNTIME_DIR=/run/user/1004`, it can't find the socket even if PulseAudio is running.

The old agent Chrome processes (PIDs 19703/19704, killed today) were launched without `XDG_RUNTIME_DIR`. They had `PULSE_SINK` and `PULSE_SOURCE` set but no socket path → every OpenAI Realtime connect attempt got `"Requested device not found"`.

New processes (41780/41781) launched with full env — confirmed:
```
XDG_RUNTIME_DIR=/run/user/1004
PULSE_SINK=agent1_speakers (or agent2_speakers)
PULSE_SOURCE=agent1_mic (or agent2_mic)
HOME=/home/agent
DISPLAY=:99
```

### @swarminged Chrome env (PID 19705, original process)
```
PULSE_SINK=swarming_playback      ← Space audio out → playback sink
PULSE_SOURCE=x_swarming_mic      ← agent audio in → swarminged's microphone
HOME=/home/agent
DISPLAY=:99
# NO XDG_RUNTIME_DIR
```
This process predates the PulseAudio crash. It still works because PulseAudio reconnection is handled by the client library — when PulseAudio came back up, Chrome reconnected automatically.

## OpenAI Realtime API — What We Learned

### Two API generations, identical-looking surface
| | Beta | GA |
|---|---|---|
| Session endpoint | `/v1/realtime` | `/v1/realtime/sessions` or `/v1/realtime/client_secrets` |
| SDP endpoint | `/v1/realtime` | `/v1/realtime/calls` |
| Key prefix | `ek_` (GA key doesn't work here) | `ek_` |
| Error if mixed | "API version mismatch. You cannot start a Realtime beta session with a GA client secret" | N/A |

Using `/v1/realtime/calls?model=gpt-4o-realtime-preview` for SDP is the only combination that works with GA ephemeral keys.

### Events on the `oai-events` data channel
| Event (send) | Effect |
|---|---|
| `session.update` | Update system instructions, voice, turn detection config |
| `response.create` | Trigger a response. Can include `instructions` override for that response only. |
| `response.cancel` | Abort current in-flight response (used by claim-token) |
| `conversation.item.create` | Inject a message into the conversation history (used to inject human speech) |

| Event (receive) | Meaning |
|---|---|
| `input_audio_buffer.speech_started` | VAD detected someone talking |
| `input_audio_buffer.speech_stopped` | VAD silence threshold hit |
| `response.audio_transcript.delta` | Streaming text of what the AI is saying |
| `response.audio_transcript.done` | Full transcript for this response turn |
| `response.done` | Response complete (may include usage stats) |

### Model and voice options
- Model: `gpt-4o-realtime-preview` (only valid option for GA as of 2026-05)
- Voices: `marin` (Swarm, warmer), `cedar` (Swarm2, deeper)

## X Space Join Flow — Complete Reference

### Why it's hard
The X Space UI has multiple states that look similar but need different interactions:

```
State 1: Home feed with a Space card
  → click the space card button → opens a mini-player peek overlay

State 2: Navigated to /spaces/ID/peek (or just /spaces/ID)
  → "Start listening" button (aria-label) → opens a modal

State 3: Modal open ([data-testid="sheetDialog"])
  → "Start listening" button INSIDE modal → actually joins

State 4: Already in another Space
  → X shows: "You cannot join this Space since you are already in a Space."
  → Must leave current Space first

State 5: Recorded Space warning
  → "Got it" dialog → must dismiss before "Request to speak" appears

State 6: Joined as listener
  → "Request to speak" button → sends request to host
  → Host sees notification on phone (unreliable)
  → OR: Host taps participant avatar → "Add as speaker" (reliable)

State 7: Accepted as speaker
  → "Unmute" button appears
  → Must click Unmute to start transmitting
```

### The speaker request notification problem
We sent 8+ speaker requests across the session. The host (user on phone) never reliably saw them. Reasons:
1. **Rate limiting** — X likely throttles repeated requests from same account
2. **Wrong Space context** — swarminged Chrome's mini-player was pointed at an old Space while the new one was being joined
3. **Notification settings** — X app notifications may not surface in-app speaker requests
4. **Workaround that always works:** Host taps the listener's avatar → "Add as speaker"

### Reliable join script (copy-paste)
```javascript
// Run via: gcloud compute ssh swarm-agent ... --command="sudo node -e \"...\""
const p = require('/home/agent/x-spaces-v2/node_modules/puppeteer-core');
const SPACE = 'https://x.com/i/spaces/SPACE_ID_HERE';
(async()=>{
  const b = await p.connect({browserURL:'http://127.0.0.1:9223', defaultViewport:null});
  const pg = (await b.pages())[0];

  // Step 0: leave any current Space
  await pg.evaluate(()=>{
    const leave=[...document.querySelectorAll('button')].find(b=>b.getAttribute('aria-label')==='Leave'&&b.offsetParent!==null);
    if(leave) leave.click();
  });
  await new Promise(r=>setTimeout(r,1500));

  // Step 1: navigate
  await pg.goto(SPACE, {waitUntil:'domcontentloaded', timeout:20000});
  await new Promise(r=>setTimeout(r,4000));

  // Step 2: click peek-level "Start listening"
  await pg.evaluate(()=>{
    const b=[...document.querySelectorAll('button')].find(b=>b.getAttribute('aria-label')==='Start listening'&&b.offsetParent!==null);
    if(b) b.click();
  });
  await new Promise(r=>setTimeout(r,2500));

  // Step 3: click modal button (or dismiss Got it)
  await pg.evaluate(()=>{
    const gotIt=[...document.querySelectorAll('span')].find(s=>s.innerText.trim()==='Got it');
    if(gotIt){gotIt.closest('button')?.click();return;}
    const modal=document.querySelector('[data-testid="sheetDialog"]')||document.querySelector('[role="dialog"]');
    if(modal){
      const btns=[...modal.querySelectorAll('button')].filter(b=>b.offsetParent!==null);
      const btn=btns.find(b=>/listen/i.test(b.innerText))||btns[btns.length-1];
      if(btn) btn.click();
    }
  });
  await new Promise(r=>setTimeout(r,2000));

  // Step 4: request to speak
  await pg.evaluate(()=>{
    const b=[...document.querySelectorAll('button')].find(b=>b.getAttribute('aria-label')==='Request to speak'&&b.offsetParent!==null);
    if(b) b.click();
  });
  console.log('Done. Accept on phone or tap avatar → Add as speaker.');
  process.exit(0);
})();
```

## Full Bug Timeline

### Bug 1 — OpenAI Realtime "API version mismatch"
**Symptom:** "You cannot start a Realtime beta session with a GA client secret"
**Cause:** SDP URL was `/v1/realtime` (beta) — GA key only works with `/v1/realtime/calls`
**Fix:** SDP URL → `https://api.openai.com/v1/realtime/calls?model=gpt-4o-realtime-preview`

### Bug 2 — "Failed to parse SessionDescription. Expect line: v="
**Symptom:** Chrome console error during SDP exchange
**Cause:** OpenAI returning a JSON error body instead of SDP — triggered by wrong model name
**Fix:** Model → `gpt-4o-realtime-preview` (not `gpt-realtime`)

### Bug 3 — Agents overtalking
**Symptom:** Both agents speak simultaneously, stepping on each other
**Cause:** No mechanism to cancel an in-flight response when the other agent starts speaking
**Fix:** Claim-token in `statusChange("speaking")` → `cancelResponse` → `response.cancel` on data channel

### Bug 4 — Agents silent after first exchange
**Symptom:** Agents spoke once, then stopped
**Cause:** No mechanism to trigger the other agent after one finishes speaking
**Fix:** `sendWhenIdle` banter loop in `textComplete` handler

### Bug 5 — Agent 2 not responding
**Symptom:** Only Agent 0 ever responded to humans or banter
**Cause:** `if (AGENT_ID !== 0) return` gate in agent2.html's `textToAgent` handler
**Fix:** Removed gate from agent2.html

### Bug 6 — "Requested device not found" on OpenAI connect
**Symptom:** Every agent page connect attempt failed with audio device error
**Cause #1:** PulseAudio not running (crashed or never started)
**Cause #2:** Chrome launched without `XDG_RUNTIME_DIR=/run/user/1004` — can't find PulseAudio socket
**Fix:** Restart PulseAudio as agent user; relaunch Chrome with XDG_RUNTIME_DIR

### Bug 7 — swarminged getting stuck in wrong Space
**Symptom:** Clicking "Start listening" would join a different Space than intended
**Cause:** Mini-player still active from previous Space; new Space navigation created a "You are already in a Space" modal
**Fix:** Click Leave before navigating to new Space URL

### Bug 8 — X Space join not working (2-click problem)
**Symptom:** "Start listening" clicked but nothing happened
**Cause:** Two separate "Start listening" elements — one opens a modal, the modal has a second button that actually joins
**Fix:** Click by position in `[data-testid="sheetDialog"]` not just by aria-label

### Bug 9 — Agents talking three.ws content despite ATL rapper system prompt
**Symptom:** Agents generating three.ws persona text all session ("Totally agree. It's like having a really good toolkit...")
**Cause:** `dc.onopen` sends `response.create` with hardcoded `instructions: "Greet the room warmly... three.ws..."` — this override instruction fires on every connect, establishes three.ws context, and OpenAI Realtime continues that persona for all subsequent responses
**Fix:** Changed dc.onopen instruction to `"Start freestyling immediately. Drop 2 bars. ATL style."` (done today)
